📄 Agent的"说明书"

放在项目根目录，Agent启动时自动读取。内容包括：

1. 身份和职责
"你是XXX项目的开发Agent，负责根据需求实现功能并通过测试"

2. 技术栈和规范
"项目使用Python 3.12 + FastAPI + PostgreSQL。代码风格遵循PEP 8"

3. 可以做的事
"可以读写src/目录下的文件，可以运行pytest，可以查询数据库"

4. 绝对不能做的事
"不得修改config/prod.yaml，不得直接操作生产数据库，不得删除migration文件"

5. 工作流程
"收到需求后：先写设计文档→实现代码→写单元测试→运行测试→通过后提PR"

6. 遇到问题时
"如果测试3次仍失败，停止并报告。遇到架构决策时，先征求人工确认。"

Hooks 和 Checkpoint
🪝

Hooks（钩子）

在关键节点插入逻辑

• 每次工具调用前：检查权限
• 每次代码生成后：自动lint
• 每次文件修改后：运行测试
• 检测到错误循环：强制终止

类比：生产线上的质检站
每个环节都过一道检查

💾

Checkpoint（检查点）

定期保存任务进度

• 每完成一个子任务存档
• 包含：已完成步骤、中间结果
  当前状态、待做任务
• 失败时可以回退到上个存档
• 长任务跨会话时恢复

类比：游戏存档
死了从存档点重来

🔒

权限控制

最小权限原则

• 文件系统：只能访问指定目录
• 网络：只能请求白名单域名
• 命令：禁止rm -rf、sudo等
• 数据：脱敏处理敏感信息

类比：公司门禁卡
不同级别不同权限

💡 Harness设计的核心原则

1️⃣ 轻量化：不要过度设计——模型升级后，复杂逻辑可能没用了
2️⃣ 可拆卸：Harness的每个组件应该可以独立替换
3️⃣ 可观测：所有Agent行为必须有日志，可追溯可审计
4️⃣ 渐进式：从最简单的Harness开始，遇到问题再加组件
5️⃣ Sensor优先：先有检测能力再有纠正能力——不能检测的问题无法修复
AI博士：记住：Start simple, add complexity only when you need it. Manus重写了5次不是因为第一次设计不好，而是因为需求和模型能力都在变。
